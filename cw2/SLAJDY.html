1. Dlaczego Programowanie Obiektowe (OOP)?

Programowanie proceduralne (C) skupia się na funkcjach operujących na danych. OOP skupia się na obiektach, które łączą dane i zachowania.

Trzy filary OOP:

  • Enkapsulacja (Hermetyzacja) – Ukrywanie stanu wewnętrznego i udostępnianie interfejsu.
  • Dziedziczenie – Tworzenie nowych klas na bazie istniejących.
  • Polimorfizm – Jednolity interfejs dla różnych typów danych.
Obiekt = Stan (Dane) + Zachowanie (Metody)

2. Od Struktur do Klas

W C++ struct i class są niemal identyczne. Różnią się jedynie domyślnym poziomem dostępu.

Struktura (struct)

  • Domyślnie: public
  • Używana głównie dla prostych paczek danych (POD - Plain Old Data).

Klasa (class)

  • Domyślnie: private
  • Używana, gdy chcemy chronić dane i wymusić interfejs.
class Date { int m_year; // Prywatne domyślnie! public: void print(); // Publiczne };

3. Specyfikatory Dostępu: public vs private

Kontrola nad tym, kto może modyfikować wnętrze naszego obiektu.

  • private: Dostępne tylko dla metod wewnątrz tej samej klasy. Chroni przed błędami logicznymi z zewnątrz.
  • public: Interfejs klasy. Widoczny dla każdego, kto posiada obiekt.
Złota zasada: Pola danych (zmienne członkowskie) powinny być zawsze prywatne.
private: int m_month { 1 }; // m_ oznacza "member" (pole klasy)

4. Gettery i Settery (Akcesory i Mutatory)

Skoro dane są prywatne, jak z nich korzystać? Przez publiczne metody.

Zalety stosowania metod zamiast bezpośredniego dostępu:

  • Walidacja: Setter może sprawdzić, czy miesiąc nie wynosi 13.
  • Tylko do odczytu: Możemy udostępnić tylko getter (brak settera = stała dla zewnątrz).
  • Abstrakcja: Możemy zmienić sposób przechowywania danych bez zmiany kodu użytkownika.
int getMonth() const { return m_month; } // Getter void setMonth(int m) { if (m >= 1 && m <= 12) m_month = m; // Walidacja! }

5. Metody Pomocnicze i Spójność Stanu

Nie każda metoda musi być publiczna. Metody prywatne pomagają organizować kod wewnątrz klasy.

Utrzymywanie inwariantów (spójności):

Obiekt nie powinien nigdy wejść w stan nielogiczny.

class Employee { std::string m_name; char m_initial; public: void setName(std::string_view name) { m_name = name; updateInitial(); // Metoda pomocnicza } private: void updateInitial() { m_initial = m_name[0]; } };

6. Konstruktory: Narodziny Obiektu

Konstruktor to specjalna metoda wywoływana automatycznie w momencie tworzenia obiektu.

  • Ma taką samą nazwę jak klasa.
  • Nie posiada typu zwracanego (nawet void).
  • Może być przeciążany (wiele konstruktorów dla jednej klasy).
class Foo { public: Foo() { /* domyślny */ } Foo(int x, int y) { /* parametryczny */ } }; Foo f1; // Woła domyślny Foo f2{ 5, 10 }; // Woła parametryczny

7. Lista Inicjalizacyjna Członków

Najważniejszy mechanizm inicjalizacji w C++. Wykonywany zanim wejdziemy do klamerek konstruktora.

Wydajność: Zapobiega domyślnej inicjalizacji pola, a następnie nadpisywaniu go wewnątrz konstruktora.
// NIEPOLECANE (Przypisanie): Foo(int x) { m_x = x; } // POLECANE (Inicjalizacja): Foo(int x) : m_x{ x } // Lista inicjalizacyjna { }

Wymagane dla: pól const, referencji oraz klas bez konstruktora domyślnego.

8. Dobre Praktyki (Best Practices)

  • Const Correctness: Każda metoda, która tylko czyta dane, powinna mieć const na końcu: int getValue() const;.
  • Nazewnictwo: Używaj m_ dla pól prywatnych. To natychmiast odróżnia je od zmiennych lokalnych i parametrów.
  • Single Responsibility: Klasa powinna odpowiadać za jedną rzecz.
  • std::string_view: Używaj jako parametrów w setterach dla lepszej wydajności (brak kopiowania).

9. Podsumowanie Wykładu

  1. Klasa to połączenie enkapsulacji danych (private) i metod operujących na nich (public).
  2. Bezpośredni dostęp do danych jest zabroniony – komunikacja odbywa się przez Gettery i Settery.
  3. Metody typu const gwarantują bezpieczeństwo – dają pewność, że stan obiektu nie zostanie zmieniony podczas odczytu.
  4. Konstruktor automatyzuje inicjalizację, a Lista Inicjalizacyjna jest najbardziej wydajnym sposobem jego implementacji.
